# Evollis — Customer Support

An AI-assisted customer-support app. Customers chat with an AI assistant; their
messages are **categorised** and then **answered** by an LLM, while human agents
manage the conversations from an inbox.

- **Client side** (`/`) — customers sign in, send messages and browse past
  conversations. Each message is categorised and answered automatically; a
  feedback control lets the customer mark a reply resolved or hand the thread to
  a human.
- **Agent side** (`/inbox`) — human operators read the full history, reply,
  set priority, assign conversations, manage labels, and filter by status.

## Tech stack

- [Next.js](https://nextjs.org) (App Router) + React
- [tRPC](https://trpc.io) for the typed API layer
- [Prisma](https://prisma.io) ORM on PostgreSQL
- [Tailwind CSS](https://tailwindcss.com)
- [OpenAI](https://platform.openai.com) for reply generation and (optionally)
  classification; a local [BERTopic](https://maartengr.github.io/BERTopic/)
  service as the free, scalable classification alternative

## Quick start

```bash
cp .env.example .env          # then set OPENAI_API_KEY (required)
npm install
npm run db:push               # apply the Prisma schema
npm run db:seed               # demo agents, clients and conversations
npm run dev                   # http://localhost:3000  (client) + /inbox (agents)
```

The login modals are pre-filled with valid demo credentials (identity is a
simple cookie — there is no password); just click **Se connecter**.

📖 **Full guide:** see [documentation/USAGE.md](documentation/USAGE.md) for a
step-by-step "How to use", configuration, and the data model.

## How a customer message is handled

1. The message is persisted and the **whole conversation** is sent to a
   classifier, which assigns a category — `BILLING` / `TECHNICAL` / `GENERAL`
   (shown in the inbox as *Facturation* / *Technique* / *Général*).
2. Based on that category, the reply is generated by OpenAI using a generic
   Evollis system prompt **plus** a category-specific system prompt.
3. The reply is posted back into the thread (a "writing…" indicator shows while
   it generates); the customer can mark it resolved or hand the thread over to a
   human agent.

### Classification providers — `CLASSIFIER_PROVIDER`

Two classifiers are implemented; the active one is chosen by an env variable:

| `CLASSIFIER_PROVIDER` | What runs | Cost | Notes |
|---|---|---|---|
| `openai` *(default)* | LLM call ([`src/server/ai/classify.ts`](src/server/ai/classify.ts)) | Billed per message | Zero setup; works as soon as `OPENAI_API_KEY` is set. |
| `bertopic` | Local semantic-clustering service ([`ml/bertopic/`](ml/bertopic/)) | Free, scalable | The production-grade path. Requires running the Python service and fitting on proprietary data for good accuracy — see [`ml/bertopic/README.md`](ml/bertopic/README.md). |

Reply generation always uses OpenAI; the provider toggle only affects how the
**category** is assigned. To switch to BERTopic, start the service in
`ml/bertopic/` and set `CLASSIFIER_PROVIDER="bertopic"` in `.env`.

## Quoi faire avec 3 jours de plus ?

Rien dans l'immédiat : la tâche m'a pris ~4h, et je ne vois pas comment
améliorer la qualité **sans bénéficier de données propriétaires**. En présence
de ces données, il aurait fallu ~6-8h pour intégrer les améliorations suivantes :

1. **Fine-tuner** le call d'API OpenAI (ou un autre modèle) de catégorisation
   sur des exemples réels de l'entreprise — pertinent dès que le volume dépasse
   500-1000 exemples, ce qui est le cas. — *~1h*
2. **Se passer des appels d'API** pour la catégorisation : avec un grand volume
   de questions (> 1000-5000), calculer les distances inter-cluster et fixer les
   seuils à xx %. Cela rend la catégorisation plus fiable **et** moins chère
   (c'est le rôle du service BERTopic, ici non entraîné faute de données). — *~1h*
3. **M'entretenir avec le responsable commercial** pour établir le playbook des
   réponses à donner selon la catégorie déterminée. — *~30min si les documents
   existent déjà*
4. **Authentification employée à l'échelle de l'organisation** (gestion de la
   création de comptes employés). La partie employée n'était pas demandée au
   départ, donc je ne suis pas allé jusqu'à construire l'infra entreprise, mais
   ce serait à faire. — *temps dépendant de l'existant*
5. **Authentification client à l'échelle de l'organisation** en connectant la
   base de données à leur CMS. S'implémente trivialement via un webhook déclenché
   à la création d'un nouveau client côté CMS, qui ajouterait ce client à la table
   du projet (ou en interrogeant directement la table de leur CMS). Faute de cette
   information, ce n'est pas implémenté. — *temps dépendant de l'existant*
6. **Récupérer les e-mails en direct** (ou les utiliser comme données de seed).
   Non fait faute d'accès, mais à prévoir. — *temps dépendant de l'existant*
